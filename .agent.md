# Agents Cloud Agent Brief

This file is the fast onboarding brief for any AI agent working in this
repository. Read it first, then read `AGENTS.md` and the roadmap docs it points
to.

## Mission

Agents Cloud is a CDK-backed autonomous AI agent platform. The product goal is a
24/7 CEO command center where a user can give high-level strategic commands and
the system delegates work to isolated agent teams.

The platform must support agents that can:

- collaborate in teams,
- run in parallel,
- delegate to specialist agents,
- perform deep research,
- create tools,
- write and test code,
- commit/open code changes,
- create documents, reports, datasets, dashboards, websites, and previews,
- use Miro as a collaboration/artifact surface,
- run each agent or team in dedicated ECS containers when isolation is needed,
- improve only through tested, auditable, human-governed promotion paths.

The user experience should feel like a CEO giving a trusted assistant a command.
The assistant plans, delegates, tracks work, asks for approval where needed, and
returns artifacts plus an executive report.

## Architecture Boundaries

Core rule: AWS owns durable execution truth.

Durable AWS plane:

- CDK owns VPC, S3, DynamoDB, ECS, Step Functions, IAM, and platform backend
  infrastructure.
- DynamoDB stores work item, run, task, event, artifact, approval, preview,
  user-runner, and membership state.
- S3 stores durable per-user/per-workspace artifacts, previews, audit records,
  and large payloads.
- ECS runs isolated workers for agents, code, builds, tests, evals, and
  previews.
- Step Functions orchestrates durable run lifecycles.

Realtime/client plane:

- AWS-native API Gateway WebSocket realtime is the primary realtime path for the
  current phase.
- Cloudflare Workers/Durable Objects stay as an alternate/fallback edge
  realtime plane until auth, replay, envelope, and client behavior match the AWS
  path.
- Neither AWS WebSocket connection state nor Cloudflare Durable Objects may be
  the only source of run/event truth.
- Next.js is the web client.
- Flutter is the desktop/mobile client.
- Clients render canonical events and validated A2UI/GenUI surfaces.

Auth/tooling plane:

- Amplify/Cognito is the current identity foundation.
- The next priority is access-code onboarding, Cognito group gates, durable
  workspace membership, and capability checks.
- OpenAI API-key/service-account auth is the production default for model calls.
- Linked Codex/ChatGPT auth is optional private/trusted-runner work only until
  policy, isolation, and reliability are validated.
- Miro/GitHub credentials must be brokered, scoped, encrypted, revocable, and
  audited.

## Next Build Slice

The next platform-critical work is tenant authorization and access-code gating:

```text
valid access code
  -> Cognito signup/confirmation
  -> platform user/org/workspace membership
  -> Cognito group assignment
  -> Control API resolveAuthContext
  -> workspace/capability checks
  -> realtime subscribe authorization
  -> scoped runner context
```

Do not enable broad Miro writes, GitHub pushes, public preview publishing, or
self-promoting specialist agents before this boundary exists.

## Implementation Principles

- Keep changes scoped.
- Preserve existing user work in the working tree.
- Prefer existing repo patterns over new abstractions.
- Prefer schema-driven contracts over ad hoc string payloads.
- Do not commit secrets, local credentials, generated outputs, build artifacts,
  or environment-specific Amplify/CDK outputs.
- Keep docs current when implementation state changes.
- Only mark checkboxes complete when code and validation prove the claim.
- Generated UI must use validated declarative packets and allowlisted catalogs,
  never arbitrary client code from an agent.

## Practical Warning

The fastest path is a boring, durable, tenant-safe vertical slice:

```text
admit user
  -> create membership
  -> authorize command
  -> create work/run
  -> schedule worker
  -> emit status
  -> write artifact
  -> stream/query status
```

Once that works, the richer CEO assistant, specialist teams, Miro, GitHub,
generated dashboards, wildcard website previews, and Cloudflare edge fallback can
plug into a real platform spine.
