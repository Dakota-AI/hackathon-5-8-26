## Artifact creation and project website delivery

Your website-preview requirement is the part most teams get wrong. The scalable answer is **not** ÔÇťone DNS record and one `ţłÇentityţłé["software","Application Load Balancer","AWS layer 7 load balancer"]ţłü` rule per project forever.ÔÇŁ `ţłÇurlţłéAWSţłéhttps://aws.amazon.comţłü` documents that an ALB supports host-header rules, but it also has a default quota of 100 non-default rules per load balancer. I would implement preview delivery in two modes. For lightweight static or mostly static sites, let the build agent write output to EFS or object storage, then serve it through a thin preview web container. For heavier interactive apps, let the build pipeline publish a dedicated preview service that joins the internal service namespace, then let the ingress router map `xyz.projects.example.com` to that task endpoint. ## Open questions and limitations

The main unresolved architectural question is how hard you want to lean on `Codex` credits versus standard API billing. Today, the docs support ChatGPT-linked usage for Codex surfaces and some trusted CI scenarios, but they still recommend API keys for programmatic automation.
