_Last updated: 2026-05-09_

## What the AWS Sample Gives Us

The sample is useful as a shape/reference, not something to copy blindly.

Useful pieces:

- Next.js App Router layout.
- Amplify client configuration from `amplify_outputs.json`.
- `@aws-amplify/ui-react` Authenticator for fast sign-in/sign-up.
- `aws-amplify` client library usage.
- `@aws-amplify/adapter-nextjs` path for server-side auth/session handling.
- Amplify Hosting build spec that deploys `.next` for a real Next app.

Less useful for this project:

- The sample's Todo AppSync/Data model is not our core backend.
- It uses Amplify Data/AppSync as the main app backend, while Agents Cloud already has CDK-owned durable backend resources.
- It demonstrates public API key auth for sample todos; Agents Cloud should use Cognito user-pool auth and a CDK Control API.

## Recommended Agents Cloud Frontend Shape

Use the template as a starting pattern for `apps/web-next`, but adapt it to this architecture:

```text
apps/web-next
  -> Next.js App Router frontend
  -> Amplify Auth / Cognito for sign-in
  -> Control API for run creation/status/events
  -> future realtime event plane for live run updates
```

Keep this boundary:

```text
Amplify owns:
  - Cognito/Auth
  - frontend hosting
  - frontend/client config

CDK owns:
  - Control API
  - DynamoDB run/task/event state
  - S3 artifacts/previews
  - ECS/Fargate runtime
  - Step Functions orchestration
```

## Why Not Copy the Template Exactly

The template creates an Amplify Data GraphQL API and Todo model. That is not the correct source of truth for this product's agent runs.

For Agents Cloud, run/task/event state should remain in the CDK DynamoDB tables because:

- Step Functions and ECS workers need direct backend access.
- The run lifecycle should be durable and platform-owned.
- The Control API can enforce authorization and hide raw table structure.
- Later Cloudflare realtime/event relay can consume the same canonical event stream.

Amplify Data may still be useful later for UI-only preferences/settings, but not as the primary run ledger.

## What We Should Build Next in the Frontend

### Phase 1: Replace Placeholder Hosting with Next.js Shell

Create a real package at `apps/web-next`:

- `package.json`
- `next.config.js` or `next.config.mjs`
- `tsconfig.json`
- `app/layout.tsx`
- `app/page.tsx`
- `app/globals.css`
- `components/ConfigureAmplifyClientSide.tsx`
- `components/AuthShell.tsx`
- `lib/amplify-client.ts`
- `lib/control-api.ts`

Initial screens:

- login/sign-up shell using Amplify Authenticator,
- authenticated dashboard shell,
- platform status cards,
- disabled/create-run CTA until Control API exists.

### Phase 2: Wire Amplify Auth Client Config

Use `amplify_outputs.json` for local development and Amplify CI output generation.

Important Next.js detail from docs:

```ts
Amplify.configure(outputs, { ssr: true });
```

Call this early from a client component imported into the root layout.

### Phase 3: Add Server-Side Auth Utilities

Add `@aws-amplify/adapter-nextjs` and create a server helper like:

```ts
createServerRunner({ config: outputs })
```

This allows server routes/middleware to read Cognito sessions safely.

### Phase 4: Control API Integration

Once `ControlApiStack` exists, add:

- `NEXT_PUBLIC_AGENTS_CLOUD_API_URL`
- `POST /runs` client helper
- `GET /runs/:runId` client helper
- `GET /runs/:runId/events` client helper

Frontend should send the Cognito id/access token to the Control API. The Control API should validate JWTs server-side.

### Phase 5: First Real Product Workflow

UI workflow:

1. User signs in.
2. User lands on dashboard.
3. User clicks Create Run.
4. Frontend calls Control API.
5. User sees run detail screen.
6. Events/status/artifacts update from durable backend state.

## Build/Hosting Recommendation

Current Amplify Hosting is green with a static placeholder generated by:

```bash
pnpm amplify:hosting:build
```

When `apps/web-next` exists, switch root `amplify.yml` from placeholder output to real Next.js build:

```yaml
version: 1
frontend:
  phases:
    preBuild:
      commands:
        - corepack enable
        - corepack prepare pnpm@10.0.0 --activate
        - pnpm install --frozen-lockfile
    build:
      commands:
        - pnpm --filter @agents-cloud/web-next build
  artifacts:
    baseDirectory: apps/web-next/.next
    files:
      - '**/*'
  cache:
    paths:
      - node_modules/**/*
      - apps/web-next/.next/cache/**/*
```

If we want static export first, use `output: 'export'` and artifact directory `apps/web-next/out`, but for authenticated dashboards and future server-side auth, normal Amplify Next.js SSR hosting is the better long-term path.

## Current Blocker for Real Auth Frontend

The project currently has Amplify Auth sandbox deployed from `infra/amplify`, while Amplify Hosting is connected to the repo root.

Before a real Auth UI deploy, choose one of these patterns:

### Option A: Keep Amplify backend in `infra/amplify`

Pros:

- Preserves clear infra package organization.
- Keeps CDK and Amplify concerns separated.

Need:

- CI/build step that generates or copies `amplify_outputs.json` into `apps/web-next`.
- Environment-specific outputs handling.
- Clear docs for sandbox vs branch backend.

### Option B: Move Amplify backend to repo-root `amplify/`

Pros:

- Matches AWS sample and Amplify Hosting expectations.
- CI can generate `amplify_outputs.json` automatically for the branch.
- Fewer custom build steps.

Cons:

- Breaks the current `infra/amplify` organization.
- Requires moving package/docs and updating workspace/scripts.

Recommendation:

Use Option A short-term only if we want to preserve structure. Use Option B if Amplify Hosting/Gen 2 branch deploy becomes the dominant frontend workflow.

Given this project already treats CDK as the durable platform owner, Option A is acceptable, but we must be intentional about output generation.

## Immediate Next Step Recommendation

Do not add Amplify Data/Todo from the sample.

Next best build sequence:

1. Build CDK `ControlApiStack` first.
2. Add `apps/web-next` Next.js shell based on the AWS template's App Router/Auth patterns.
3. Configure Amplify Auth client setup with `ssr: true`.
4. Add dashboard screens that are ready for Control API.
5. Switch Amplify Hosting from placeholder build to real `apps/web-next` build.

Reason:

The frontend becomes much more useful once it has a real API endpoint to create/query runs. Without the Control API, the Auth UI can log users in but cannot do the core product action yet.
